V1 Build Checklist

Ayudita Document Decoder

A simple print checklist for building the first version: one document comes in, Ayudita stores it, reads it, explains it in Spanish, and shows the proof.

North Star

Ayudita should help a family understand one scary letter in under two minutes without making up anything the letter did not say.

Phase 1 - Product Focus

1. Lock the V1 scope

V1 should only decode one document at a time. It should not try to be a full memory app yet.

Why it matters: A smaller product is easier to test with real families.
Done when: everyone agrees what V1 will and will not do.

2. Keep raw documents as the source of truth

Every photo or PDF must be saved before the AI tries to read it. If AI fails, the original file is still safe.

Why it matters: We can rebuild AI results later, but we cannot rebuild a lost document.
Done when: the file is stored before extraction starts.

Phase 2 - Supabase Foundation

3. Create the V1 database tables

Add tables for documents, facts, explanations, document text, and user questions.

Why it matters: These tables match the new trust-first architecture.
Done when: Supabase has the V1 schema running.

4. Create private document storage

Create a private Supabase bucket called raw-documents for original files.

Why it matters: User letters are private and sensitive.
Done when: files upload to a private bucket.

Phase 3 - Web Upload Flow

5. Add document upload in the dashboard

Let the user upload a photo or PDF from the web app. Start with images first if needed.

Why it matters: This lets us test before WhatsApp is fully connected.
Done when: a user can upload a real letter.

6. Show document status

Show clear states like received, extracted, explained, failed, and pending review.

Why it matters: The user and team should always know where the document is.
Done when: every document has a visible status.

Phase 4 - AI Extraction

7. Build the extraction stage

The vision model reads the image and returns only structured JSON. It does not write the final answer.

Why it matters: Separating reading from explaining makes the system safer.
Done when: extraction creates facts with source text.

8. Store facts with proof

Every important fact needs a value, a source quote, and a label like verbatim, computed, inferred, or unknown.

Why it matters: Ayudita should earn trust by showing evidence.
Done when: facts are stored with provenance.

Phase 5 - AI Explanation

9. Build the explanation stage

The explanation model should only see extracted facts. It should never see the image directly.

Why it matters: It cannot explain facts that were not extracted.
Done when: explanations are based only on facts.

10. Use the six required sections

The answer must include what the letter is, why it was sent, important dates, what to do, evidence, and what could not be determined.

Why it matters: This gives users a calm, predictable answer every time.
Done when: every explanation follows the same structure.

Phase 6 - Review Dashboard

11. Turn the inbox into a review queue

The dashboard should show documents waiting for review, not general life memories.

Why it matters: The first version needs human review before replies go out.
Done when: pending documents are easy to review.

12. Show the explanation and the evidence

The dashboard should show the Spanish explanation next to the source text that supports it.

Why it matters: This is what makes Ayudita feel trustworthy.
Done when: reviewers can compare answer and proof.

13. Add review actions

Add buttons for approve, flag, ask for clearer photo, and regenerate.

Why it matters: Real letters are messy, and the team needs control.
Done when: a reviewer can decide the next step.

Phase 7 - WhatsApp

14. Receive WhatsApp images

When a user sends a photo, the webhook should download the image and save it first.

Why it matters: WhatsApp is the easiest place for families to use Ayudita.
Done when: WhatsApp photos appear in the review queue.

15. Send approved replies back

After human review, Ayudita should send the Spanish explanation back through WhatsApp.

Why it matters: The user gets help in the same place they sent the letter.
Done when: approved explanations can be sent by WhatsApp.

Phase 8 - Safety And Testing

16. Handle blurry or unreadable photos

If the photo is not clear, Ayudita should say it cannot read enough and ask for a better photo.

Why it matters: Saying “I do not know” is safer than guessing.
Done when: bad photos fail gracefully.

17. Log follow-up questions

Save what users ask after the explanation, because those questions show what to build next.

Why it matters: Real users will write the phase-two roadmap for us.
Done when: questions are saved in user_questions.

18. Test with 10 real families

Test with real letters. Watch especially for wrong dates, missing evidence, and unclear replies.

Why it matters: The best product signal comes from real people using it.
Done when: 10 users have tried it and given feedback.
Build rule: Do not add memory, timelines, agents, vector search, or multi-document reasoning before this checklist works well. The decoder comes first. The bigger personal cloud can be built later from the raw documents we save today.
